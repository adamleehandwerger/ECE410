# m4 Testbenches

## Interface Coverage

There are two distinct test interfaces used in m4 — **not all tests go through Wishbone**:

| Testbench | Interface | What it exercises |
|-----------|-----------|-------------------|
| `tb_wb_cosim.py` | **Wishbone only** | Full Caravel wrapper — all stimulus via WB register map (CONTROL, STATUS, NUM_SAMPLES, ALPHA_WR, etc.); the testbench acts as the host MCU |
| `dv_run.sh` / `svm_wb_test.c` | **Wishbone only** | Caravel DV framework — management SoC firmware drives the same WB registers over the RTL bus |
| `svm_ram_latency_tb.sv` | **Direct RTL port** | Instantiates `svm_compute_core` directly, no wrapper, no Wishbone — drives `start`, `ram_rdata`, `num_samples` pins to isolate and verify the `RAM_LATENCY` wait-state logic |

The 300-sample accuracy cosim (`tb_wb_cosim.py`) is the primary validation and exercises
the full Wishbone path end-to-end. The RAM_LATENCY unit test is intentionally wrapper-free
so that wait-state timing can be verified against the core in isolation.

## Comparison

| Column | Implementation | Description |
|--------|---------------|-------------|
| Left   | **Numba Q6.10** | Python JIT, parallel, exact hardware model (same Q6.10 arithmetic) |
| Right  | **ASIC RTL**    | svm_compute_core driven via Wishbone register interface |

All ASIC predictions are produced by `tb_wb_cosim.py`, which exercises
`user_project_wrapper` exclusively through its Wishbone registers:
- Feature words written one-by-one via FIFO_DATA (0x30000000 + 0x00)
- Classification triggered via CONTROL register (0x30000004)
- Result read from GPIO[2:0] (io_out[2:0] = class_out_r)

## WB Interface Protocol (tb_wb_cosim.py)

Per-sample classification sequence through the register interface:

1. **One-time config** (before first sample):
   - Write `NUM_SAMPLES = 1`
   - Write `NUM_SV[0..4]` (sklearn SV counts, capped at 50 per class)
   - Write `PARAM_WR` to set γ = 0.25 (Q6.10)
   - Write `CONTROL = 0x0A` (vbatt_ok=1, kern_ready=1) — wait 6 clocks for 2-FF sync

2. **Per sample** (×300 test samples):
   - Write `CONTROL = 0x0B` (start pulse; auto-clears after 1 clock)
   - Write `FIFO_DATA` × 256 (one 16-bit Q6.10 feature word per write)
   - Poll `io_out[3]` (GPIO[3] = svm_done) until high
   - Capture `io_out[2:0]` (GPIO[2:0] = class_out) at same cycle

3. **After all samples**: write predictions to `asic_preds.csv`

## SV RAM Model (off-chip interface)

The svm_compute_core reads support vectors from an off-chip RAM via:
- `io_out[25]` = sv_ram_ren (read enable)
- `sv_ram_addr_w[17:0]` = {sv_global_idx[9:0], feat_dim[7:0]} (internal signal)
- `la_data_in[15:0]` = sv_rdata (Python drives this after each ren pulse)

The `sv_ram_model` coroutine in `tb_wb_cosim.py` models 1-cycle latency (RAM_LATENCY=1,
the default). The `RAM_LATENCY` RTL parameter can be raised for slower physical SRAMs
(e.g., RAM_LATENCY=3 for the IS61WV51216 async SRAM at 40 MHz). Use
`svm_ram_latency_tb.sv` to verify wait-state correctness for a given latency setting:

```bash
# Compile and run the RAM_LATENCY unit test (iverilog only, no Caravel wrapper needed)
iverilog -g2012 -DSIMULATION -o /tmp/svm_lat_tb.out \
    ../../rt1/svm_compute_core.sv svm_ram_latency_tb.sv
./tmp/svm_lat_tb.out
# Expected: PASS: 10/10 beats classified, done=1, no sticky errors
```
